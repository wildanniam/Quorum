# Soroban Spike Notes

Date: 2026-06-07

## Tooling Findings

- Local Stellar CLI: `stellar 26.0.0`.
- Local Rust: `rustc 1.95.0`.
- Installed contract targets: `wasm32v1-none` and `wasm32-unknown-unknown`.
- `stellar contract init` scaffolds a workspace with `contracts/*`, per-contract tests, and release profile optimized for small WASM.
- The CLI 26 scaffold currently sets `soroban-sdk = "25"`. The generated contracts pass `cargo test` and `stellar contract build`.

## Library Decision

OpenZeppelin Stellar Contracts provides audited NFT utilities, but Quorum MVP needs a narrow, non-transferable pass with custom event ownership and one-pass-per-wallet checks. For the hackathon MVP, implement a minimal custom `QuorumPassNFT` contract and keep OpenZeppelin as a post-MVP hardening/reference option.

Reasons:

- transfer must be disabled by design;
- minting must be restricted to `QuorumCore`;
- pass lookup must be optimized around `event_id + owner`;
- custom contract is smaller and easier to test under hackathon time.

## Implementation Direction

- Keep two contracts: `quorum_core` and `quorum_pass_nft`.
- Use `stellar contract init` workspace layout.
- Use SDK version generated by local CLI unless build/test fails.
- Use `u32` split percentages scaled as basis points inside contracts, while UI keeps human percentages.
- Use explicit tests for duplicate claim/purchase, capacity, split totals, check-in, and withdraw accounting.
